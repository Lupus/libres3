Error messages interpretation

  2. LibreS3 / s3cmd: WARNING: Retrying failed request: /path/file Errno -2 Name or service not known

Installation

  1. If you want to build from source you can use:

    apt-get install ocaml-native-compilers camlp4 camlp4-extra libssl-dev libpcre3-dev git make m4
git clone https://github.com/enospc/libres3.git
cd libres3/
./configure && make && make install
libres3_setup

  2. If you want to build from source you can use:

    yum install ocaml ocaml-camlp4-devel ocaml-camlp4 ocaml-compiler-libs ocaml-runtime pcre-devel openssl-devel make m4 ncurses-devel git
git clone https://github.com/enospc/libres3.git
cd libres3/
./configure && make && make install
libres3_setup

  3. By default CentOS/RHEL blocks incoming traffic to all ports except for 22. You need to allow traffic to the ports used by Sx and LibreS3.

    iptables -I INPUT -p tcp --dport 443 -j ACCEPT
iptables -I INPUT -p tcp --dport 8443 -j ACCEPT
service iptables save

    If you want to also allow access over plaintext HTTP (not recommended), you need to open the following ports:

    iptables -I INPUT -p tcp --dport 80 -j ACCEPT
iptables -I INPUT -p tcp --dport 8008 -j ACCEPT
service iptables save

Error messages interpretation

  1. LibreS3 provides an S3 REST interface to access your data.

    The server will detect attempts to access the root of the cluster from a browser and serve you the “Your LibreS3 node is up and running” page instead.

    If you want to access your LibreS3 cluster from a browser, you need to enable public buckets, and access the public bucket URL, not the cluster root.

  2. The S3 protocol requires you to create an A record and a wildcard record for your LibreS3 host. Assuming your /etc/libres3/libres3.conf has s3_host=s3.mydomain.com and the IP address of your LibreS3 node is 1.2.3.4, you need to add these two entries to your DNS zone:

    s3.mydomain.com. A 1.2.3.4
*.s3.mydomain.com. A 1.2.3.4

    As an example, if you run:

    s3cmd ls

    s3cmd will resolve s3.mydomain.com and ask 1.2.3.4 to provide the list of available buckets.

    If you run:

    s3cmd ls s3://mybucket

    s3cmd will resolve mybucket.s3.mydomain.com and ask 1.2.3.4 to list the content of that bucket.

    The same applies to other S3 clients.

    If you need to do a quick test and you don’t want to change your DNS zone, you can add multiple entries to the /etc/hosts file of the client machine where you are running s3cmd:

    1.2.3.4 mybucket.s3.mydomain.com mybucket2.s3.mydomain.com s3.mydomain.com
